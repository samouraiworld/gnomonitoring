## Part 1 — Go Code Quality

### CRITICAL

**C1. `WatchValidatorAlerts` goroutine has no panic recovery**
File: `gnovalidator_realtime.go:375`

`WatchValidatorAlerts` starts a bare `go func()` with no `defer recover()`. Every other long-running goroutine in the codebase (`CollectParticipation`, etc.) uses panic recovery. If the CTE query panics (e.g. nil gorm.DB), the entire process crashes.

```go
go func() {
    // missing: defer func() { if r := recover(); r != nil { log.Printf(...) } }()
    for { ... }
}()
```

**Fix:** Add `defer recover()` consistent with all other monitoring goroutines.

---

**C2. `WatchValidatorAlerts` calls `GetThresholds()` twice per cycle — TOCTOU on threshold values**
File: `gnovalidator_realtime.go:431, 445`

Line 431 uses `GetThresholds().WarningThreshold` in the SQL `WHERE missed >= ?`. Line 445 calls `t := GetThresholds()` again inside the per-validator loop. If an admin updates thresholds between the two calls (mid-cycle), a sequence row can pass the SQL filter but fail the Go-level `switch` check (or vice versa), causing missed or double alerts.

**Fix:** Call `GetThresholds()` once before the SQL query and reuse the same snapshot throughout the cycle.

**W6. `SendResolveAlerts` — zero-row scan indistinguishable from `participated = false`**
File: `gnovalidator_realtime.go:581-593` (pre-existing, not fixed by this PR despite plan targeting this function)

```go
var countparticipated int
db.Raw(`SELECT participated FROM daily_participations WHERE ... AND block_height = (?+1)`, ...).Scan(&countparticipated)
if countparticipated == 0 { continue }
```

When `end_height+1` is not yet indexed, `Scan` returns no rows and `countparticipated` stays `0` — same result as `participated = false`. RESOLVED alerts are silently skipped for validators whose recovery block hasn't been indexed yet.

**Fix:** Use `SELECT COUNT(*) ... WHERE ... AND participated = 1`.

**W13. `FetchChainHealthSnapshot` called in Telegram handler has no timeout**

**Fix:** Wrap `FetchChainHealthSnapshot` with `context.WithTimeout(ctx, 10*time.Second)` or add individual timeouts per RPC call.

## Part 2 — Security

### WARNING

**S1. Unvalidated `v=` period value stored in session state**
File: `telegram/validator.go:1754`

The raw `v=` value from the period-selection callback is stored directly into `CmdState.Period` without whitelist validation:

```go
state.Period = params["v"]
```

It later flows to `getPeriodParams(period)` which rejects unknown values — so the DB is not queried with arbitrary input. However: the error message from `getPeriodParams` includes the raw value and is sent to the Telegram chat via `fmt.Sprintf("❌ %v", err)` without HTML-escaping.

**Fix:**
```go
validPeriods := map[string]bool{
    "current_week": true, "current_month": true,
    "current_year": true, "all_time": true,
}
if !validPeriods[params["v"]] {
    _ = EditMessageTelegramWithMarkup(token, chatID, messageID, "⚠️ Invalid period.", nil)
    return
}
state.Period = params["v"]
```

---

**S2. Raw chain IDs encoded into callback data without `&`/`=` sanitization**
File: `telegram/validator.go:1952`

`encodeCmdCallback` does not escape `&` or `=` within values. Chain IDs come from the config file (trusted), but a malformed chain ID containing `&a=cx` would produce a callback that triggers a cancel action in any chat that taps the button. The downstream `validateChainID` check at line 1891 prevents IDOR between chains, but the parsing ambiguity is a silent failure mode.

**Fix:** Strip or reject `&` and `=` from chain IDs before encoding, or add a `sanitizeCallbackValue` step in `encodeCmdCallback`.

---

**S3. `FormatAlertsLast24h` renders unescaped monikers/addresses as HTML in Telegram**
File: `health.go:467-473`

`FormatAlertsLast24h` is plain text (used in daily reports sent to Discord/Slack/Telegram). For the Telegram path, this same string is sent with `parse_mode: "HTML"`. A validator moniker stored in `alert_logs` containing `<b>`, `<code>`, or `<a href=...>` tags would render as HTML markup in Telegram messages.

Source of monikers is on-chain (valopers realm or genesis), which requires governance — realistic attack surface is limited but non-zero.

**Fix (preferred):** Create a companion `FormatAlertsLast24hHTML` that applies `html.EscapeString` to moniker and address fields, and use it for Telegram. Keep `FormatAlertsLast24h` for Discord/Slack plain-text paths.

---

**S4. DB error strings forwarded verbatim to Telegram users**
File: `telegram/validator.go:2066, 2072, 2078, 2098, 2104, 2110, 2116, 667`

GORM error messages can include table names, column names, and SQL fragments. These are sent directly to the Telegram chat via `fmt.Sprintf("❌ %v", err)`.

**Fix:** Log the full error server-side and return a generic user-facing message:
```go
log.Printf("[cmd] action error cmd=%s action=%s: %v", state.Command, state.Action, err)
return "❌ An error occurred. Please try again later.", nil
```

---

### INFO

**S5.** `CmdState.Action` stored from raw callback without whitelist enforcement. Currently safe (all actions fall through to "Nothing to execute" on unrecognized values, and values are HTML-escaped in confirmation messages). Low risk given Telegram's server-signed callback data.

**S6.** `decodePeriod` returns `""` for unknown `r=` codes, which silently defaults to `periodDefault` in all command branches. Not exploitable — existing behavior is safe.

**S7.** `resendHours` / `silenceWindow` formatted as `fmt.Sprintf("-%d hours", n)` and passed as GORM bind parameters (not string-interpolated into SQL). SQLite receives it as a literal modifier string. Safe — noted because the pattern could raise concern in future audits.

**S8.** `startSearchStateCleanup()` spawns a cleanup goroutine per `BuildTelegramHandlers` call. In tests or multi-call scenarios this creates redundant goroutines. No data corruption; `sync.Once` guard would be cleaner.
