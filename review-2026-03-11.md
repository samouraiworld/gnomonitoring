# Performance Review — 2026-03-11

> Full codebase review of the Gnomonitoring backend.
> Focus: performance, stability, resource usage.
> Scope: DB/SQL, GovDAO/alerts, goroutines/concurrency, API/Telegram.

---

## Priority 0 — Bugs (not performance, but will cause incidents)

### ~~0.0 CORS preflight (OPTIONS) ne renvoie pas Access-Control-Allow-Origin~~ — FIXED

**File:** `internal/api/api.go` — 15 occurrences

`EnableCORS` accepte `r *http.Request` en variadic pour le matching multi-origine. Tous les handlers normaux appelaient `EnableCORS(w, r)` correctement, mais **tous les blocs `case http.MethodOptions`** appelaient `EnableCORS(w)` sans `r` — l'origine n'était jamais lue, le header `Access-Control-Allow-Origin` n'était pas écrit sur les preflights. Le browser bloquait donc toutes les requêtes cross-origin malgré le CORS correct sur les vraies routes.

**Fix:** `sed -i 's/EnableCORS(w)$/EnableCORS(w, r)/g'` → 15 occurrences corrigées.

### ~~0.1 Race conditions~~ — FIXED

`lastProgressHeight` → `atomic.Int64`, `alertSent`/`restoredNotified` → `atomic.Bool`, `lastRPCErrorAlert`/`lastProgressTime` protected by `timeMu sync.Mutex`.

**File:** `internal/gnovalidator/gnovalidator_realtime.go` lines 17–23

`lastProgressHeight`, `lastProgressTime`, `alertSent`, `restoredNotified`, and `lastRPCErrorAlert` are global variables written and read from the `CollectParticipation` goroutine with **no mutex or atomic protection**. Running `go test -race` will flag these.

**Consequences:** wrong alert state → duplicate or missed alerts; in extreme cases, partial writes on non-atomic types.

**Fix:** Wrap in a `sync.Mutex` or use `sync/atomic` for the bool/int64 fields.

---

### ~~0.2 WebSocket govdao no reconnect~~ — FIXED

Exponential backoff retry loop added in `WebsocketGovdao`: starts at 2s, doubles on each failure, caps at 60s.

**File:** `internal/govdao/govdao.go` ~line 185

`WebsocketGovdao` returns on any read error. The caller (`StartGovDAo`) does not retry. Once the WebSocket drops (network blip, server restart), proposal monitoring stops permanently until the process is restarted.

**Fix:** Wrap the entire connect+read loop in a retry loop with exponential backoff.

## Priority 1 — Critical (fix before next release)

### 1.1 No graceful shutdown (all goroutines) — goroutines run forever

**Files:** `main.go`, `gnovalidator/gnovalidator_realtime.go`, `govdao/govdao.go`

Every long-running goroutine (`CollectParticipation`, `WatchNewValidators`, `WatchValidatorAlerts`, `StartGovDAo`, `StartProposalWatcher`) uses an infinite `for` loop with no context or stop channel. `main.go` ends with `select {}` and no signal handling.

On SIGTERM the process dies instantly without flushing the DB write batch or closing the SQLite connection cleanly — risk of WAL corruption.

**Fix:**

```go
// main.go
ctx, stop := signal.NotifyContext(context.Background(), syscall.SIGINT, syscall.SIGTERM)
defer stop()

go gnovalidator.StartValidatorMonitoring(ctx, db)
go govdao.StartGovDAO(ctx, db)
// ...

<-ctx.Done()
// give goroutines 10s to finish
shutdownCtx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
defer cancel()
<-shutdownCtx.Done()
```

Pass `ctx` to every function that spawns goroutines, check `ctx.Done()` in each loop.

### ~~1.2 panic() on DB error~~ — FIXED

Replaced `panic(err)` with `log.Printf + return err` in `MsgTelegram` and `MsgTelegramAlert`.

**File:** `internal/telegram/telegram.go` lines 227, 252

`MsgTelegram` and `MsgTelegramAlert` call `panic(err)` on `GetAllChatIDs` failure. Since these are called from goroutines, an unexpected DB hiccup crashes the whole process.

**Fix:** Replace `panic(err)` with `log.Printf` + `return err`.

### 1.3 daily_participations grows unbounded — no retention policy

There is no pruning job. After 1 year the table will have ~50M rows (assuming ~100 blocks/hour × 24 × 365 × 15 validators). Every full-table-scan query becomes proportionally slower. Disk usage grows without limit.

**Fix:** Add a nightly job:

```sql
DELETE FROM daily_participations WHERE date < DATE('now', '-90 days');
```

Or a GORM scheduled task. Expose the retention period as a config key.

---

### 1.5 CalculateConsecutiveMissedBlocks loads entire table

**File:** `internal/gnovalidator/metric.go`

The alert-check function loads the full `daily_participations` table for every validator on every tick (every 20s). With a large history this is O(N) memory and a full table scan.

**Fix:** Rewrite using SQL window functions or a bounded query:

```sql
SELECT addr, participated
FROM daily_participations
WHERE block_height > (SELECT MAX(block_height) FROM daily_participations) - 200
ORDER BY block_height DESC;
```

---

### 1.4 Sequential webhook dispatch blocks the alert loop

**File:** `internal/gnovalidator/metric.go`

Discord/Slack webhooks are dispatched synchronously inside the alert loop. One slow or timed-out HTTP call freezes the entire 20-second cycle for all validators.

**Fix:** Dispatch each webhook in a goroutine with a bounded worker pool (max 10) and a per-request context with a 5-second timeout:

```go
const webhookTimeout = 5 * time.Second

func sendWebhookAsync(url, payload string) {
    go func() {
        ctx, cancel := context.WithTimeout(context.Background(), webhookTimeout)
        defer cancel()
        // do the POST with ctx
    }()
}
```

---

### 1.5 No HTTP timeouts on webhook/Slack/Discord outbound calls

**Files:** `internal/gnovalidator/metric.go`, `internal/govdao/govdao.go`

`SendDiscordAlert` and `SendSlackAlert` use `http.DefaultClient` which has no timeout. A hung external service freezes the goroutine permanently.

**Fix:** Use a package-level client with a 10s timeout (same pattern as `telegramHTTPClient`):

```go
var alertHTTPClient = &http.Client{Timeout: 10 * time.Second}
```

---

### 1.6 strftime() in WHERE prevents index use on date column

**File:** `internal/gnovalidator/metric.go`

Queries filter `strftime('%Y-%m-%d', date) = ?` which wraps the indexed column in a function — SQLite cannot use the index and performs a full scan.

**Fix:** Store and filter dates as ISO strings, use range queries:

```sql
WHERE date >= '2026-03-01' AND date < '2026-03-02'
```

### ~~1.6 InsertChatID blocks polling loop~~ — FIXED

Moved to a fire-and-forget goroutine. The govdao last-proposal send is also now async inside that goroutine.

**File:** `internal/telegram/telegram.go` line 365

`InsertChatID` is a synchronous DB write called directly in `StartCommandLoop`, before command dispatching. Any DB latency (lock contention, WAL checkpoint) stalls processing of all incoming Telegram updates.

**Fix:** Fire-and-forget the DB write in a goroutine (it's idempotent — it's an upsert):

```go
go func(chatID int64) {
    database.InsertChatID(db, chatID, TypeChatid)
}(up.Message.Chat.ID)
```

---

### ~~1.7 http.Post() no connection reuse~~ — FIXED

Added `var alertHTTPClient = &http.Client{Timeout: 10 * time.Second}` in `fonction.go`. `SendDiscordAlert` and `SendSlackAlert` now use it.

**File:** `internal/fonction.go` lines 51, 68 and others

`SendDiscordAlert` and `SendSlackAlert` use `http.Post()` which internally uses `http.DefaultClient` but creates a new transport per connection. This means **no TCP connection reuse** across webhook dispatches.

**Fix:** Declare a package-level client (same pattern as `telegramHTTPClient`):

```go
var alertHTTPClient = &http.Client{Timeout: 10 * time.Second}
```

---

### 1.8 Unbounded scheduler goroutines at startup

**File:** `internal/scheduler/scheduler.go`

`StartAll(db)` creates **one goroutine per row in `hour_reports`**. `StartAllTelegram(db)` creates **one goroutine per active Telegram chat**. With 1000 users + 500 chats = 1500 goroutines sleeping until their scheduled time.

Each goroutine is ~4 KB of stack. 10K users = ~40 MB just in scheduler goroutines. More critically, these all wake up and query the DB at approximately the same time each morning → thundering herd.

**Fix:** Replace one-goroutine-per-user with a single scheduler goroutine using a min-heap of (nextRun, userID), waking up only when the next report is due.

**File:** `internal/telegram/validator.go` lines 447–502

`subscribe on all`: 1 query to list validators + 1 UPDATE per validator = 100+ queries.
`unsubscribe all`: 1 query to list subs + 2 queries per sub = 100+ queries.

**Fix:** Single batch statement:


### ~~2.6 HTTP server no timeouts~~ — FIXED

Replaced `http.ListenAndServe` with `http.Server{ReadTimeout: 15s, WriteTimeout: 30s, IdleTimeout: 90s}`.

**File:** `internal/api/api.go` ~line 1057

Vulnerable to slow-loris and idle connection accumulation.

**Fix:**

### 2.7 Unbounded goroutine spawning in Telegram command loop

**File:** `internal/telegram/telegram.go` lines 353, 388, 390

Every callback and command spawns a new goroutine with `go h(...)`. Under burst traffic (many users, many commands), this creates thousands of goroutines competing for the DB.

**Fix:** Use a bounded worker pool:

```go
var handlerSem = make(chan struct{}, 20) // max 20 concurrent handlers

func runHandler(fn func()) {
    go func() {
        handlerSem <- struct{}{}
        defer func() { <-handlerSem }()
        fn()
    }()
}
```

---

### 2.8 timer.After() leak in scheduler

**File:** `internal/gnovalidator/gnovalidator_report.go` lines 38, 65

`time.After(wait)` creates a timer that is never cancelled if the reload channel fires first. In a system with many user reports, these timers accumulate.

**Fix:** Use `time.NewTimer` + explicit `Stop()`:

```go
timer := time.NewTimer(wait)
select {
case <-timer.C:
    sendReport()
case <-reload:
    timer.Stop()
    return
}
```

**File:** `internal/database/db_metrics.go` ligne 252

### 3.1 Cache stampede risk in Telegram metrics cache

**File:** `internal/telegram/validator.go`

All 5 cache entries share the same 45s TTL. They all expire at once after the first cache fill, sending 5 simultaneous DB queries.

**Fix:** Add ±10s jitter to TTL:

```go
func cacheTTLWithJitter() time.Duration {
    jitter := time.Duration(rand.Intn(20)-10) * time.Second
    return cacheTTL + jitter
}
```

Also consider increasing TTL to 2–5 minutes — metrics don't change per block.

---

### 3.2 Missing Content-Type headers in API handlers

**File:** `internal/api/api.go` (all handlers)

JSON responses don't set `Content-Type: application/json`. Some browsers and reverse proxies may misinterpret the response.

**Fix:** Add a helper:

```go
func writeJSON(w http.ResponseWriter, v any) {
    w.Header().Set("Content-Type", "application/json")
    json.NewEncoder(w).Encode(v)
}
```

---

### 3.3 Missing HTTP 500 on handler errors

**File:** `internal/api/api.go` multiple handlers

On DB errors, handlers call `return` without setting a status code — client receives HTTP 200 with an empty body. Pattern seen in `Getblockheight`, `GetAddrMonikerHandler`, and others.

**Fix:** Always use `http.Error(w, msg, http.StatusInternalServerError)` on errors.

---

### 3.4 300+ DB queries per 20s alert cycle

**File:** `internal/gnovalidator/metric.go`

The alert loop queries `GetAllChatIDs` + `GetTelegramValidatorSub` + webhook dispatch inside the inner validator loop. For 20 validators and 15 telegram subscribers = 300+ queries every 20s.

**Fix:**

- Preload subscriber list once per tick, reuse across all validators
- Cache the active-sub map in memory, invalidate on subscription change

### 3.5 Lock held during slow MonikerMap iteration

**File:** `internal/gnovalidator/gnovalidator_realtime.go` ~lines 217–235

`MonikerMutex.RLock()` is held while comparing all entries (logging, comparisons). Blocks writes from `InitMonikerMap`.

**Fix:** Snapshot the map under lock, then process the copy outside the lock:

```go
MonikerMutex.RLock()
snapshot := maps.Clone(MonikerMap)
MonikerMutex.RUnlock()
// process snapshot without lock
```

### 3.6 ResolveAddrs queries wrong table

**File:** `internal/database/db_telegram.go` ~line 205

`ResolveAddrs` runs `SELECT DISTINCT moniker, addr FROM daily_participations WHERE addr IN (?)` — a DISTINCT scan on the large participation table.

#### F1 — CRITICAL : Dev mode `X-Debug-UserID` permet l'usurpation d'identité complète

**Fichier :** `internal/api/api.go`, lignes 26–32

Quand `DevMode = true`, `authUserIDFromContext` retourne n'importe quelle valeur fournie dans le header `X-Debug-UserID` sans aucune validation. Si le header est absent, la fonction retourne silencieusement `"local-dev-user"` et toutes les opérations d'écriture passent sans token.

De plus, en mode dev, les routes protégées ne sont **pas** wrappées dans le middleware Clerk (lignes 951–957) — il n'y a aucun fallback.

**Exploit :** `curl -X DELETE "http://host/alert-contacts?id=1" -H "X-Debug-UserID: victim_clerk_id"`

**Fix :** S'assurer que `DevMode` ne peut pas être activé en production (build tag ou variable d'environnement contrôlée par l'infra). Ne jamais servir un endpoint d'écriture sans authentification, quel que soit le mode.

#### F2 — HIGH : SQL via `fmt.Sprintf` dans `GetTimeOfBlock` et `GetTimeOfAlert`

**Fichier :** `internal/database/db_metrics.go`, lignes 335–363

```go
query := fmt.Sprintf(`SELECT DISTINCT date FROM daily_participations WHERE block_height = %d;`, numBlock)
query := fmt.Sprintf(`SELECT DISTINCT sent_at FROM alert_logs WHERE start_height = %d and end_height =%d;`, numBlock, numBlock)
```

Ces requêtes contournent la paramétrisation de GORM. `numBlock` est un `int64` (pas d'injection directe possible aujourd'hui), mais ce pattern bypass la défense en profondeur et deviendra une injection réelle si le type du paramètre évolue.

**Fix :** Remplacer par `db.Raw(query, numBlock)` avec des placeholders `?`.

#### F3 — HIGH : Injection de nom de table dans `UpdateMonitoringWebhook`

**Fichier :** `internal/database/db.go`, lignes 172–175

```go
stmt := fmt.Sprintf("UPDATE %s SET url = ?, ... WHERE user_id = ? AND id = ?", tablename)
```

Le paramètre `tablename` est interpolé directement dans le SQL. Les appelants actuels passent des littéraux hardcodés, mais la fonction est exportée et accepte n'importe quelle chaîne.

**Fix :** Remplacer par un `switch` validant `tablename` contre les deux valeurs autorisées ; retourner une erreur pour toute autre valeur.

#### F4 — MEDIUM : Même injection de nom de table dans `GetWebhookByID`

#### F5 — HIGH : Fan-out de toutes les alertes vers TOUS les webhooks de TOUS les utilisateurs

**Fichier :** `internal/fonction.go`, lignes 104–106, 276–278, 312–314

`SendAllValidatorAlerts`, `SendResolveValidator` et `SendInfoValidator` récupèrent **tous** les webhooks sans filtre `WHERE user_id = ?`. Toutes les alertes de tous les validateurs sont envoyées à tous les webhooks de tous les utilisateurs, sans respect de la propriété.

#### F6 — HIGH : `MentionTag` injecté sans sanitisation dans les messages Discord/Slack

**Fichier :** `internal/fonction.go`, lignes 134 et 174


`mention_tag` est stocké depuis l'input utilisateur sans validation. Un utilisateur malveillant peut stocker `everyone` ou `here` pour déclencher un mention broadcast Discord, ou injecter du contenu arbitraire.

#### F7 — MEDIUM : `id_webhook` dans `InsertAlertContact` non vérifié comme appartenant à l'appelant

#### F8 — MEDIUM : Double verrou `timeMu` — deadlock silencieux / DoS

**Fichier :** `internal/gnovalidator/gnovalidator_realtime.go`, lignes 79–93

`sync.Mutex` en Go n'est pas réentrant. Si `timeMu.Lock()` est appelé deux fois de suite sans `Unlock()` intermédiaire, le goroutine se bloque indéfiniment — arrêt silencieux de la collecte de blocs.

**Fix :** Chaque section critique doit avoir exactement un `Lock` et un `Unlock`. Supprimer tous les doublons.

#### F12 — LOW : `GetUserById` renvoie un 500 au lieu d'un 404 quand l'utilisateur n'existe pas

**Fichier :** `internal/database/db.go`, lignes 225–235 / `internal/api/api.go`, lignes 376–383

`db.First()` retourne `gorm.ErrRecordNotFound`. Le handler vérifie `if user == nil` mais la fonction retourne `nil, gorm.ErrRecordNotFound` — le handler tombe dans `err != nil` et renvoie un 500 avec le message GORM exposé. Combiné au K2, cela permet l'énumération d'utilisateurs.

**Fix :** Dans `GetUserById`, tester `errors.Is(err, gorm.ErrRecordNotFound)` et retourner `nil, nil` dans ce cas.
