# Multi-Chain Backend Code Review
**Date:** 2026-03-19
**Branch:** `feat/multi-chain`
**Reviewer:** Claude Sonnet 4.6
**Scope:** Full backend multi-chain implementation (Phases 1–3)

---

## Summary

The multi-chain implementation is structurally sound and demonstrates careful engineering: all database queries are correctly scoped by `chain_id`, per-chain goroutines are isolated, and the in-memory state maps (MonikerMap, alertSent, etc.) have been properly promoted to two-level maps with correct mutex coverage. Test coverage of the multi-chain invariants is solid. Several issues exist that warrant attention before production deployment, the most critical being an SSRF vulnerability in the webhook URL dispatch and a `log.Fatalf` inside a goroutine.

---

## Strengths

- **Clean chain isolation in the database layer.** Every table that stores chain-specific data (`daily_participations`, `alert_logs`, `addr_monikers`, `govdaos`, `telegram_validator_subs`, `telegram_hour_reports`) carries a `chain_id` column, all raw SQL queries consistently pass `chain_id` as a bind parameter, and the unique indexes are compound `(chain_id, ...)`. Cross-chain data leakage in the DB layer is not possible by construction.

- **Correct concurrent in-memory state management.** All five multi-chain in-memory maps (`MonikerMap`, `lastProgressHeight`, `alertSent`, `restoredNotified`, `chatChainState`) are promoted to two-level maps keyed by `chainID` and each is protected by its own `RWMutex`. Accessor functions (`GetMonikerMap`, `SetMoniker`, `IsAlertSent`, etc.) encapsulate locking correctly, and the pattern is uniform throughout the codebase.

- **Robust backfill pipeline.** `BackfillParallel` uses a clean worker-pool pattern with a dedicated closer goroutine and a single-writer collector, correctly respecting the SQLite 999-variable limit via `flushBatch`/`flushChunk` chunking. Block data is never written from multiple goroutines simultaneously.

- **Idempotent schema migrations.** `ApplyMultiChainMigrations` checks for column existence via `PRAGMA table_info` before running any `ALTER TABLE`, making it safe to re-run. The `telegram_hour_reports` table recreation is handled via a copy-insert-drop-rename sequence, which is the only valid approach in SQLite.

- **Comprehensive multi-chain integration tests.** The `multichain_integration_test.go` file explicitly verifies data isolation across chains for participation data, last-stored height, in-memory moniker maps, and alert logs. The `db_migrations_test.go` file validates every new column and index. All tests pass cleanly with `go vet` reporting no issues.

## Issues Found

### CRITICAL

**C1 — SSRF: webhook URLs are dispatched without any validation**
**Files:** `internal/fonction.go:134` (`SendDiscordAlert`), `internal/fonction.go:150` (`SendSlackAlert`)
**Details:** The `alertHTTPClient` sends HTTP POST requests to whatever URL is stored in the `webhook_validators` or `webhook_gov_daos` tables. There is no allowlist, no scheme check (`http://` vs `https://`), and no hostname validation. An authenticated user can register a webhook URL pointing to an internal service (e.g., `http://169.254.169.254/latest/meta-data/`, `http://localhost:6060/debug/pprof/`) and receive data from it through the alert mechanism. In a self-hosted cloud deployment this is a classic SSRF attack surface.

---

**C2 — `log.Fatalf` called inside a goroutine terminates the entire process on a per-chain error**
**Files:** `internal/gnovalidator/gnovalidator_realtime.go:586`, `internal/govdao/govdao.go:297`, `internal/govdao/govdao.go:452`
**Details:** `StartValidatorMonitoring` calls `log.Fatalf` if the RPC client cannot be created for a chain. Because `startChainMonitoring` in `main.go` spawns each chain in its own goroutine with `go startChainMonitoring(...)`, a single misconfigured chain will kill the entire service. The same pattern appears in `ExtractTitle` and `ExtractProposalRender`. The correct behaviour is to log the error and return (or restart with backoff), not to terminate the process.

---

### HIGH

**H1 — `LoadConfig` panics with an index-out-of-range if no chains are enabled (but chains map is non-empty)**
**File:** `internal/fonction.go:81`
**Details:** When `Config.DefaultChain` is set to a non-existent or disabled chain, the fallback logic accesses `EnabledChains[0]` directly. If no chains have `enabled: true`, `EnabledChains` is empty and this panics. The `log.Fatalf` guard on line 65 only fires when `len(Config.Chains) == 0`; a config with chains all set to `enabled: false` passes that check but causes a panic at line 81 or 84.

---

**H2 — Alert deduplication logic bug in `WatchValidatorAlerts`: `skipped=1` filter is inverted**
**File:** `internal/gnovalidator/gnovalidator_realtime.go:339-358`
**Details:** The first deduplication check reads:
```sql
SELECT COUNT(*) FROM alert_logs
WHERE chain_id = ? AND addr = ? AND level = ?
AND start_height = ?
AND skipped = 1
```
The comment says "Check if alert was recently sent." However the `skipped` flag means the alert was *not* actually dispatched (it was a duplicate row written to avoid re-sending). When `count > 0`, the code skips sending — but then also writes another `skipped = false` row (line 354). This logic is correct only if `skipped = 1` rows are exactly the sentinel "we already acted on this start_height". The naming is confusing and the inverted meaning (`skipped=1` meaning "already processed / do not resend") should be documented explicitly. The current behaviour produces a `skipped=false` row every polling interval for an ongoing miss run, which inflates the `alert_logs` table.

---

**H3 — `GetMonikerMap` returns a live reference to an internal map slice, not a copy**
**File:** `internal/gnovalidator/gnovalidator_realtime.go:600-607`
**Details:**
```go
func GetMonikerMap(chainID string) map[string]string {
    MonikerMutex.RLock()
    defer MonikerMutex.RUnlock()
    if m, ok := MonikerMap[chainID]; ok {
        return m   // <-- returns the inner map directly
    }
    return make(map[string]string)
}
```
The returned map is the actual inner map stored in `MonikerMap[chainID]`. Callers in `BackfillParallel` and `CollectParticipation` read this map from multiple goroutines after the lock is released. If `WatchNewValidators` calls `InitMonikerMap` (which calls `SetMoniker`) concurrently, the inner map can be mutated while a worker goroutine is iterating over it via `for addr, mon := range monikerMap`. In Go, concurrent map iteration + write is a data race. The fix is to return a shallow copy of the map inside the lock.

---

**H4 — `SendAllValidatorAlerts` sends an empty `fullMsg` to Slack for WARNING level**
**File:** `internal/fonction.go:229-263`
**Details:** In the `"slack"` branch of `SendAllValidatorAlerts`, the `if level == "CRITICAL"` block sets `fullMsg` and then contains a nested `if level == "WARNING"` block that can never be reached (it is inside the `CRITICAL` block). The `WARNING` case for Slack therefore calls `SendSlackAlert` with `fullMsg = ""`. Discord WARNING works because it is a separate branch at the same level.

### MEDIUM

**M1 — `GetAlertLog` with `all_time` period crashes on an empty `alert_logs` table**
**File:** `internal/database/db_metrics.go:54-75`
**Details:** When no alert logs exist for the chain, `MIN(sent_at)` and `MAX(sent_at)` return SQL NULL. The code unconditionally calls `time.Parse(layout, minS.String)` where `minS.String` is `""` for a NULL value, which returns a parse error and bubbles up as a 500 to the API caller. The `all_time` path should handle the empty-table case gracefully (returning an empty slice).

---

**M2 — `StartProposalWatcher` and `StartGovDAo` are not associated with any chain-specific context**
**File:** `main.go:72`, `govdao/govdao.go:536`
**Details:** `StartProposalWatcher` is launched once globally and iterates all `govdaos` rows, correctly looking up the chain config per row. However, `StartGovDAo` is called per-chain but `WebsocketGovdao` has no reconnection timeout on the initial `connection_init`/`start` messages — a WriteJSON failure is only logged, execution continues into the read loop, and no reconnection is attempted until the next `ReadMessage` fails. This can leave the WebSocket stuck in a zombie state silently receiving nothing.

---

**M3 — `http.Get` in `InitMonikerMap` uses the default transport with no timeout**
**File:** `internal/gnovalidator/valoper.go:131`
**Details:** `GetGenesisMonikers` uses `http.Get(url)` with the default `http.DefaultClient`, which has no timeout. The genesis endpoint (`/genesis`) returns a large JSON payload (the full genesis file). A slow or stalled RPC node will block the goroutine indefinitely during startup. A dedicated `http.Client` with a timeout (e.g., 30 seconds) should be used here.

---

**M4 — `ValidateChainID` accepts disabled chains**
**File:** `internal/fonction.go:116-121`
**Details:** `ValidateChainID` only checks that the chain key exists in `Config.Chains`; it does not check that `chain.Enabled == true`. An API caller can therefore scope a webhook to a disabled (but configured) chain and receive alerts for it if that chain is later enabled. Given that `GetChainIDFromRequest` uses `ValidateChainID`, this inconsistency also means API clients can query metrics for a disabled chain and receive empty results rather than a clear "chain not enabled" error.

---

**M5 — `InsertDB` in `db_init.go:270` uses `log.Fatalf` instead of returning an error**
**File:** `internal/database/db_init.go:270`
**Details:** Inside `InitDB`, when `gorm.Open` fails, the code calls `log.Fatalf` directly rather than returning the error. `InitDB` is typed to return `(*gorm.DB, error)` — the `log.Fatalf` path bypasses the caller's error handling entirely. The `main.go` caller already handles the returned error with its own `log.Fatalf`, making the inner fatal redundant and confusing.

---

**M6 — `CheckProposalStatus` logs a spurious error message after a successful update**
**File:** `internal/govdao/govdao.go:525`
**Details:**
```go
log.Printf("❌ Failed to update govdao status id=%d: %v", p.Id, err)
// update GovDao (explicit WHERE to handle id=0)
if err := db.Model(...).Update("status", "ACCEPTED").Error; err != nil {
    log.Printf("❌ Failed to update govdao status id=%d: %v", p.Id, err)
}
```
Line 525 logs a failure error unconditionally before the update is even attempted. This is a copy-paste error: `err` at that point is `nil` (the `SendInfoGovdao`/`MsgTelegram` errors are silently discarded by the surrounding `if err :=` scoping). Every successful proposal acceptance will produce a misleading error log entry.

### LOW

**L1 — `SendSlackAlert` swallows errors and always returns nil**
**File:** `internal/fonction.go:150-165`
**Details:** `SendSlackAlert` logs HTTP errors but returns `nil` unconditionally. Callers in `SendResolveValidator` and `SendInfoValidator` cannot detect Slack delivery failures. This is inconsistent with `SendDiscordAlert`, which correctly propagates errors.

---

**L2 — `SheduleUserReport` always uses `internal.Config.DefaultChain` instead of the user's configured chain**
**File:** `internal/gnovalidator/gnovalidator_report.go:41`
**Details:**
```go
SendDailyStatsForUser(db, internal.Config.DefaultChain, &userID, nil, loc)
```
Webhook-based user daily reports are sent for the default chain regardless of which chain the user's webhook is scoped to. The scheduler does not receive a `chainID` parameter. For a single-chain deployment this is not observable, but in a multi-chain deployment a user subscribed only to `chainB` will receive a report for `chainA` (the default chain).

---

**L3 — `GetStatusofGovdao` and `GetLastExecute` reference the wrong table name**
**File:** `internal/database/db_telegram.go:302, 322`
**Details:** Both functions query `FROM govdaos` but the GORM model is `Govdao` which GORM pluralizes to `govdaos`. The table is actually named `govdaos` (confirmed by AutoMigrate), so this works at runtime, but the hardcoded name conflicts with the GORM model's table name convention and will silently break if the model is ever renamed.

---

**L4 — Prometheus `StartPrometheusServer` uses `http.ListenAndServe` on the default mux**
**File:** `internal/gnovalidator/Prometheus.go:75`
**Details:** `http.Handle("/metrics", ...)` registers on `http.DefaultServeMux`, which is then served by `http.ListenAndServe`. If any other package in the process registers a handler on `http.DefaultServeMux`, it will be served on the metrics port. A dedicated `http.ServeMux` should be used for the metrics server.

---

**L5 — `startSearchStateCleanup` goroutine is started multiple times if `BuildTelegramHandlers` is called more than once**
**File:** `internal/telegram/validator.go:98`
**Details:** `startSearchStateCleanup` starts a new background ticker goroutine every time `BuildTelegramHandlers` is called. In the current `main.go` it is called once, so there is no leak today, but there is no guard against multiple calls. Adding a `sync.Once` around the cleanup goroutine would be a simple safeguard.

---

**L6 — `flag.Bool("disable-report", ...)` is registered but `EnableDebug` is never initialised**
**File:** `internal/flag.go:8,12`
**Details:** `EnableDebug *bool` is declared in the `var` block but `flag.Bool` for it is never called in `InitFlags()`. Any code attempting to dereference `EnableDebug` will panic.

## Recommendations

1. **Implement webhook URL validation before dispatching alerts (fixes C1).** Add a validation step in `InsertMonitoringWebhook` and `InsertWebhook` that checks the URL scheme is `https` and the hostname is not a private IP range, loopback address, or link-local address. Alternatively, maintain an allowlist of accepted hostname suffixes.

2. **Replace `log.Fatalf` inside goroutines with error returns and restart logic (fixes C2, M5).** `StartValidatorMonitoring` should return an error to its caller rather than calling `log.Fatalf`. In `main.go`, wrap each per-chain goroutine with a restart loop using exponential backoff (following the existing `doWithRetry` pattern).

3. **Return a shallow copy of the inner map in `GetMonikerMap` (fixes H3).** Inside the `RLock` block, copy the map contents into a new `map[string]string` before returning. This is a one-line change with negligible cost given that moniker maps are small (tens of entries).

4. **Fix the empty `alert_logs` edge case for `all_time` period (fixes M1).** Before calling `time.Parse`, check `minS.Valid && maxS.Valid`. If the table is empty for the chain, return an empty slice with no error.

5. **Add a request timeout to the genesis HTTP client and fix the Slack WARNING message path (fixes M3, H4).** Use a dedicated `http.Client{Timeout: 30 * time.Second}` in `GetGenesisMonikers`. In `SendAllValidatorAlerts`, move the Slack WARNING message construction to the same level as the CRITICAL branch (outside the `if level == "CRITICAL"` block).

## Conclusion

The multi-chain architecture is well-designed and correctly implemented at the database and concurrency layers. The chain isolation invariants are provably correct from the schema design, the compound unique indexes, and the consistently parameterised queries. The in-memory state management follows established patterns from the codebase and adds no new race conditions.

The most urgent issues to address before production are the SSRF risk in webhook dispatch (C1), the `log.Fatalf` inside per-chain goroutines that can take down the entire service on a single chain failure (C2), the Slack WARNING empty-message bug (H4), and the `GetMonikerMap` data race under concurrent backfill + moniker refresh (H3). The remaining items are lower-priority quality improvements. No SQL injection vulnerabilities were found — all queries use parameterised placeholders throughout.
