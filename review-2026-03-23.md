## 1. Summary

**Overall status**: NEEDS FIXES

**Key findings**:
- **Goroutine leak in timeout mechanism**: When `StartMetricsUpdater` times out a chain update, the inner goroutine running `UpdatePrometheusMetricsFromDB` is never cancelled and continues executing indefinitely. There is no `context.Context` to signal cancellation.
- **Stale metric values on validator churn**: Prometheus `GaugeVec` metrics are never `Reset()` before update. If a validator disappears from the active set between cycles, its old metric values remain forever with the last-known value.
- **`AlertsTotal` uses Gauge for a cumulative counter**: The metric `gnoland_alerts_total` is described as "Total number of alerts sent (cumulative)" but is implemented as a `GaugeVec`. While it is set from a DB `COUNT(*)` each cycle (so it does reflect the correct value), the naming with `_total` suffix conventionally implies a Counter in Prometheus. This will confuse users of Grafana/alerting rules.
- **`StartPrometheusServer` is wrapped in an extra `go` in main.go**: The function already launches `http.ListenAndServe` in a goroutine internally, but main.go calls it with `go gnovalidator.StartPrometheusServer(...)`, creating a needless double-nesting.
- **Solid test coverage**: 6 tests covering chain isolation, multi-chain, Phase 1/2/3 metrics. All pass.

**Risk assessment**: MEDIUM

## 2. Detailed Findings

### 2.1 Concurrency

| # | Finding | Severity | Location |
|---|---------|----------|----------|
| C1 | **Goroutine leak on timeout** -- `StartMetricsUpdater` spawns a goroutine for each chain update with a 2-minute timeout via `select` + `time.After`. When the timeout fires, `wg.Done()` is called and the cycle moves on, but the inner goroutine calling `UpdatePrometheusMetricsFromDB` keeps running. On every subsequent cycle, a new leaked goroutine may accumulate if the chain is consistently slow. The correct pattern is to pass a `context.WithTimeout` and check `ctx.Err()` in the DB layer or between phases. | P0 | Prometheus.go:366-378 |
| C2 | **No panic recovery on the innermost goroutine** -- The goroutine at line 368 (`go func() { ... UpdatePrometheusMetricsFromDB ... }`) has no `defer recover()`. If a panic occurs inside that goroutine (e.g., nil pointer in a DB result), it will crash the entire process. The parent goroutine has recovery, but that only covers panics in the `select` block, not in the child. | P1 | Prometheus.go:368-374 |
| C3 | **`sync.WaitGroup` used correctly** -- Each chain gets `wg.Add(1)` before the goroutine launch, and `defer wg.Done()` is the first defer in the goroutine. This is correct. | OK | Prometheus.go:356-384 |
| C4 | **`sync.Once` for registration** -- `Init()` correctly uses `initOnce.Do()` to prevent double-registration panics. Tests call `Init()` safely multiple times. | OK | Prometheus.go:143-161 |
| C5 | **No direct MonikerMap access** -- The metrics code resolves monikers via SQL JOINs with `addr_monikers`, not via the global `MonikerMap`. No `MonikerMutex` concerns here. | OK | metric.go, db_metrics.go |

## 3. Recommendations

### P0 -- Must fix before merge

1. **Stale metrics on validator churn** -- Add `Reset()` calls at the start of `UpdatePrometheusMetricsFromDB` for all per-validator GaugeVec metrics:
   ```go
   ValidatorParticipation.Reset()
   MissedBlocks.Reset()
   ConsecutiveMissedBlocks.Reset()
   // ... etc
   ```
   Note: `Reset()` clears ALL label combinations across all chains. If chain updates run in parallel, one chain's Reset could clear another chain's freshly-set values. Solutions: (a) serialize updates (remove the WaitGroup parallelism), or (b) use `DeletePartialMatch(prometheus.Labels{"chain": chainID})` (requires prometheus/client_golang >= v1.12.0) to reset only the target chain's metrics. Option (b) is preferred.

2. **Goroutine leak on timeout** -- Replace the `time.After` pattern with `context.WithTimeout`. Pass the context into `UpdatePrometheusMetricsFromDB` and propagate it to DB calls via `db.WithContext(ctx)`. When the context is cancelled, GORM will abort in-flight queries. Without this, timed-out goroutines accumulate and hold DB connections.

### P1 -- Fix soon

3. **Add panic recovery to inner goroutine** (Prometheus.go:368) -- The goroutine spawned for the timeout mechanism has no `defer recover()`. A panic inside `UpdatePrometheusMetricsFromDB` will crash the process.

4. **`OperationTimeMetricsaddr` full table scan** -- Apply the same two-step pattern: fetch MAX(block_height) first, then limit the range (e.g., last 30 days or last 10K blocks) instead of scanning all history for `participated = 0`.

5. **`GetFirstSeen` full table scan** -- This only needs to run infrequently (validator first-seen dates rarely change). Consider caching results or limiting the scan to validators seen in recent blocks, then falling back to full scan only for new validators.

6. **`InsertAddrMoniker` missing `chainID`** -- Pre-existing bug. The function should accept and set `chainID`.

### P2 -- Polish

7. **Rename `UptimeMetricsaddr` / `OperationTimeMetricsaddr`** to follow Go naming conventions (e.g., `UptimeMetricsByAddr`).

8. **Use `%w` for error wrapping** in `OperationTimeMetricsaddr`, `UptimeMetricsaddr`, `GetFirstSeen` to allow proper error chain inspection.

9. **Remove double `go` on `StartPrometheusServer`** -- Either remove the internal goroutine (make the function blocking) and keep the `go` in main.go, or keep the internal goroutine and remove the `go` in main.go.

10. **Reduce logging verbosity** -- Consider logging at DEBUG level or summarizing the update in a single line per chain (e.g., "Updated 8 metrics for betanet: 25 validators, height=12345").

11. **Use `time.Ticker` instead of `time.Sleep`** for more predictable 5-minute cycles.

12. **Document the 200-block window** for consecutive missed blocks in the metric Help string.

### P3 -- Future improvements

13. **Add covering indexes** with `participated` and `tx_contribution` columns for date-range queries.

14. **Deduplicate MAX(block_height) calls** across `GetActiveValidatorCount` and `GetAvgParticipationRate`.

15. **Move struct types** (`ValidatorStat`, `MissedBlockStat`, `ConsecutiveMissedStat`) from Prometheus.go to metric.go.

16. **Extract date parsing helper** for FirstSeen timestamps.
